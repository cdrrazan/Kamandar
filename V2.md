# V2 — Multi-provider roadmap

> Status: **design / roadmap only.** No code in `kamandar.rb` has changed
> for this document. This is the plan for making kamandar work with GitLab
> and other project managers (Jira, Linear) while keeping it stdlib-only,
> serverless, and single-file-friendly.

## Goal

Today kamandar is GitHub-only: the engine consumes raw GitHub GraphQL node
hashes, and the network layer (`Kamandar::GitHub`) speaks the GitHub GraphQL
API. V2 should let the *same* terminal/browser surfaces and the *same*
classification logic run against:

- **GitLab** — merge requests + issues on a GitLab board.
- **Jira** — issues/sprints via the Jira REST API.
- **Linear** — issues via Linear's GraphQL API.

…without rewriting the engine or surfaces for each.

## Guiding principles (unchanged from V1)

- **Stdlib only.** Every provider must talk to its API with `net/http` + `json`.
- **Serverless.** No server, no OAuth dance — tokens/PATs via env vars.
- **Engine stays pure.** Classification and time math never learn provider
  names.
- **Surfaces never re-query or re-classify.** They consume the buckets hash.

## The core insight: a normalized work-item model

V1's engine already classifies *display-shaped* data into buckets. The coupling
is that the engine reaches into **GitHub-shaped** node hashes
(`pr.dig("commits", "nodes")`, `latestOpinionatedReviews`, etc.). V2 breaks that
coupling by inserting a **Provider** layer that maps each backend onto a small,
provider-neutral model the engine understands.

```
Provider (GitHub | GitLab | Jira | Linear)
    │  fetches + normalizes
    ▼
Normalized model (ReviewItem, WorkItem)
    │
    ▼
Engine (pure classification + time math)   ← unchanged across providers
    │
    ▼
Buckets (plain hash)
    │
    ▼
Surface (terminal | browser)               ← unchanged across providers
```

### Proposed normalized structs

Two plain hashes/structs, provider-neutral:

```ruby
# A reviewable change (PR / MR).
ReviewItem = {
  id:, number:, title:, url:, repo:,
  draft:            true/false,
  created_at:       Time,
  handoff_events:   [Time, ...],   # review-requested events + pushes + created
  reviewer_actions: [               # opinionated only; comments excluded
    { state: "APPROVED"|"CHANGES_REQUESTED", at: Time }, ...
  ],
  has_reviewer:     true/false,     # request count OR any request/action seen
  review_requested_to_me: true/false
}

# A board item (issue/card).
WorkItem = {
  id:, number:, title:, url:, repo:,
  type:        "Issue"|"Task"|...,
  assignees:   ["login", ...],
  status:      "Todo",             # normalized single-select status name
  iteration:   { start_date: Date, duration_days: Int } | nil
}
```

The current bucket-#4 logic maps cleanly onto `ReviewItem`:

- `handoff_at` = `max(handoff_events)`
- `reviewer_last_action_at` = `max(reviewer_actions.at)`
- `ball_on_reviewer?`, `stale?`, `forgot_reviewer?` operate on the struct, not
  on GitHub JSON.

This is the main refactor: change the engine's PR predicates to read the
normalized `ReviewItem` instead of GitHub node hashes. Time math
(`days_since`), the project filter (`assigned_not_started`), iteration logic,
and URL parsing are already provider-neutral and move over unchanged.

## The Provider contract

A tiny duck-typed interface — one module per backend:

```ruby
module Provider
  # Returns { owed: [ReviewItem], mine: [ReviewItem] }
  def fetch_reviews(config); end

  # Returns { items: [WorkItem], iterations: [iteration] | nil }
  def fetch_board(config); end

  # Parse a backend board URL -> provider-specific board handle
  def parse_board_url(url); end
end
```

`Kamandar::GitHub` becomes `Provider::GitHub` implementing this contract by
wrapping today's GraphQL queries and adding a normalize step (GitHub node hash →
`ReviewItem`/`WorkItem`). The CLI picks a provider via config.

### Provider selection

```
PROVIDER / --provider   default: github   one of: github | gitlab | jira | linear
```

Per-provider auth, all via env (no new flags required):

| Provider | Token var | Identity var | Board/host var |
|---|---|---|---|
| github | `GITHUB_TOKEN` | `GH_LOGIN` | `PROJECT_URL` |
| gitlab | `GITLAB_TOKEN` | `GITLAB_LOGIN` | `GITLAB_URL`, `GITLAB_HOST` (self-managed) |
| jira | `JIRA_TOKEN` (+ email) | `JIRA_ACCOUNT_ID` | `JIRA_HOST`, `JIRA_BOARD_ID` |
| linear | `LINEAR_TOKEN` | `LINEAR_USER` | `LINEAR_TEAM` |

## Per-backend mapping notes

### GitLab

- **Reviews owed**: GitLab GraphQL `currentUser.reviewRequestedMergeRequests`
  (or REST `GET /merge_requests?reviewer_username=<me>&state=opened`).
- **WIP**: authored MRs with `draft: true` (GitLab's `Draft:`/`WIP:` flag).
- **Gone quiet (#4)**: GitLab has no first-class "opinionated review" object.
  Map:
  - *handoff* ← MR `created_at`, last commit `committed_date`
    (`merge_request.commits`), and "reviewer assigned" resource events
    (`resource_milestone`/`reviewer` events via the Notes/Events API).
  - *reviewer action* ← MR **approvals** (`approvals` / approval events) as
    `APPROVED`, and a "requested changes"/unapprove event as
    `CHANGES_REQUESTED`. GitLab's review model is weaker here, so document the
    approximation.
- **Board (#3)**: GitLab issue boards expose lists (columns) as labels, not a
  single-select "Status" field. Map the **list/label** name to `status`, and
  GitLab **iterations** (a native feature) to the iteration model.

### Jira

- REST v3 (`/rest/api/3/search` with JQL). No PR concept natively — bucket #4
  and "reviews owed" only apply if PRs are linked via the dev-panel; otherwise
  the review buckets are empty and #3 (board) is the value.
- **#3**: JQL `assignee = currentUser() AND statusCategory = "To Do"` maps
  directly; `status` ← issue status name; iteration ← active **sprint**
  (`sprint in openSprints()`).

### Linear

- GraphQL. `issues(filter: { assignee: { isMe: true }, state: { type: { eq:
  "unstarted" } } })` for #3; Linear **cycles** map to the iteration model.
- No PR review model — review buckets empty unless GitHub/GitLab is also linked.

## Surfaces & buckets — no change

The buckets hash and both surfaces (terminal, browser, HTML security guarantee)
stay exactly as in V1. Per-provider differences are absorbed entirely in the
Provider layer's normalize step. The bonus *forgot-reviewer* bucket simply
stays empty for providers without a review-request concept.

## Testing strategy

- Keep `test_kamandar.rb`'s engine tests but feed them **normalized `ReviewItem`/
  `WorkItem`** fixtures instead of GitHub JSON (mechanical edit; the assertions
  don't change).
- Add a `test_providers.rb` with **canned API JSON fixtures** per provider
  (captured once, scrubbed of secrets) asserting that each provider's normalize
  step produces the expected `ReviewItem`/`WorkItem`. This keeps provider
  mapping covered with zero network.

## Migration plan (incremental, each step shippable)

1. **Extract the normalized model + provider interface**; make
   `Provider::GitHub` produce `ReviewItem`/`WorkItem`; switch the engine
   predicates to read the structs. Tests updated to normalized fixtures. *(No
   user-visible change.)*
2. **Add `PROVIDER`/`--provider` plumbing** (still only `github`).
3. **Add `Provider::GitLab`** with its own fixtures + tests.
4. **Add Jira / Linear** as board-only providers (review buckets degrade
   gracefully to empty).
5. **Docs**: per-provider setup tables in `README.md`, auth notes in
   `SECURITY.md`.

## Open questions

- Should multiple providers be **merged into one view** (e.g. GitHub reviews +
  Jira board) rather than chosen one-at-a-time? The Provider contract supports
  it; the CLI would accept a comma list and the engine would concatenate
  normalized items. Deferred until single-provider is solid.
- Self-managed GitLab/Jira hosts need a configurable base URL — already in the
  table above; just needs validation.
- Rate limiting / pagination differs per backend; the board fetch already
  paginates for GitHub and the pattern generalizes.
